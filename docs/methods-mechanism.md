# Methods mechanism — design rationale

## The problem this solves

Applied research projects accumulate operational rules that
aren't in any textbook: how *this engagement* defines an
electronics entrant, which products it counts as tradable, what
threshold floors it imposes on a sample. The rule is
project-internal — defensible, but specific. Researchers will
come back to it, modify it, and need to re-implement it
consistently across notebooks.

Without scaffolding, the failure modes are:

- **Rule lives in the script.** The implementation IS the spec.
  A second analyst, asked "what's our entrant rule?", reads
  `build_06_electronics_cohort.py` line by line. Six months
  later, the rule has drifted across three scripts and no one
  notices.
- **Rule lives in a notebook narrative.** A markdown cell
  describes the rule prose-style. Looks readable, but the prose
  and the code can diverge silently — the reader trusts the
  prose; the chart was computed from the code.
- **Rule lives in a brainstorm doc.** A
  `brainstorms/cohort-rule.md` explains the v1 design. Then v2
  ships, the brainstorm is not updated, and a year later the
  brainstorm describes a rule that no longer runs anywhere.

The methods convention sits between these. Each method gets a
sub-folder; `rule.md` carries the operational spec; the
diagnostic counts at the bottom (re-generated by the
implementing script) are the headline anchor that proves the
rule and the doc are in sync.

## Why one sub-folder per method

Considered and rejected: a flat folder of `<method>.md` files
(parallel to `data_sources/`). Methods grow heterogeneous
appendages — a CSV of edge-case exclusions kept for
transparency, a notebook tying the rule to its diagnostic
outputs, a PDF of the underlying codebook the rule references.
A flat folder forces either accepting clutter (PDFs at the
root) or scattering the appendages (codebook in
`data_sources/`, exclusions in `output/`). A sub-folder keeps
the method's full materials together.

The convention: the sub-folder name IS the handle.
`electronics_entry/`, not `methods/electronics_entry_rule.md`.
A researcher who knows the method's name finds the folder;
from the folder, the materials are obvious.

## Why no top-level INDEX

Considered and rejected. Unlike `data_sources/` (where one
source spawns multiple files and an INDEX disambiguates), each
method is its own folder. The directory listing IS the index —
`ls methods/` returns the project's full method list with one
line per method. A separate INDEX would duplicate this and rot.

If a project grows past ~10 methods, that's a signal the
engagement scope has drifted, not that the convention needs an
INDEX. Refocus the engagement.

## Why the seven required sections

The shape comes from pilot use. Each section earns its place:

- **Source.** Without it, the rule can't be reproduced. Lists
  the table, schema, year range, key columns. Pre-computed
  columns (e.g. `export_rca` from Atlas) are flagged so a
  reader doesn't unwittingly recompute them.
- **Rule.** The operational spec. Numbered steps, exact
  thresholds, exact codes. A reader can re-implement from this
  alone.
- **Why this version.** The genealogy. v1 → v2 changes are
  load-bearing — they shift headline counts, sometimes by 2x.
  Documenting *why* prevents the next session from "fixing" a
  v2 back to v1's behavior.
- **Exclusions.** Project-internal rules typically drop a list
  of codes, countries, or periods. Each exclusion paired with
  rationale (re-export hub, micro-jurisdiction, schema break).
  Transparency CSVs preserved under `output/` are referenced.
- **Edge cases.** Where the rule's plain-English statement is
  ambiguous and a deliberate call was made — NULL values,
  censored trajectories. Documenting the call freezes the
  decision.
- **Known limitations.** What the rule *doesn't* handle.
  Pre-period coverage gaps, schema breaks, deflator
  simplifications. Phrased as "accept as known" — caveats,
  not bugs to fix. Also serves as text for chart captions.
- **Diagnostic counts.** The numbers that prove the rule is
  in force. From the implementing script. Cross-tabs that
  catch regressions in the rule.

A method missing any section is a smell — the rule isn't yet
operationally codified, and the script is probably ahead of
the spec.

## Why diagnostic counts are headline anchors

The cross-cutting principle in
`docs/audience-and-philosophy.md` (Principle 9) names a
headline anchor as a concrete value the documented procedure
produces. For methods, the diagnostic counts are exactly that:
re-running the implementing script reproduces the counts; if
they drift, either the rule changed (and the doc needs
updating) or the data changed (and the rule may need
re-validation).

The pairing with `analytical-commit-format` is what makes this
work: a commit that updates the implementing script carries
`Run:` and `Out:` lines pointing at the script and its outputs.
`git log -- methods/<method>/` then traces every change to the
rule and every regeneration of its counts. The audit trail is
self-contained.

## Why versioning is in-place + decision records

Considered and rejected: separate v1 and v2 files (`rule_v1.md`,
`rule_v2.md`). Tempting for genealogy, but researchers reading
`rule.md` need to see the *current* rule, not all historical
versions. Git preserves the history; the doc should read as
truth-of-the-day.

The pattern: edit `rule.md` in place, bump the heading to vN+1,
rewrite `Why this version` to describe what changed. For
load-bearing methodology shifts (the kind that change headline
counts by >10% or that a peer reviewer would push back on),
pair the edit with a `decisions/YYYY-MM-DD_<method>-vN+1.md`
decision record. The rule.md cross-links to the decision; the
decision cross-links to the rule.

The two artifacts have distinct jobs:

- `rule.md` says **what** to do, with **diagnostic counts**
  that prove the rule.
- `decisions/...md` says **why**, with **alternatives
  rejected**.

A peer reviewer asking "why this rule?" reads the decision; an
analyst asking "what's the rule today?" reads `rule.md`.

## Boundary with `decisions/` and `wiki/concepts/`

The boundary is the load-bearing ambiguity in this convention.
Restated:

- `decisions/` — peer-reviewable methodology calls, dated,
  never edited. One per call. Light pre-registration.
- `methods/<method>/rule.md` — operational rules with
  diagnostic counts, edited in place as they evolve. One per
  recurring rule.
- `wiki/concepts/<slug>.md` — distilled domain claims with
  citations. Public knowledge. One per concept worth a wiki
  page.

The cleanest test: would another researcher writing the *same*
diagnostic from scratch arrive at the same rule by reading
domain literature? If yes, it's a `wiki/concepts/` page. If no
— the rule is project-internal — it's a
`methods/<method>/rule.md`. If the call is contestable enough
to defend in review, it's *also* a `decisions/` record
cross-linked from the rule.md.

In practice, a method is usually two or three of these: a
`wiki/concepts/` page on the broader concept (the Balassa RCA
index, with citations), a `methods/electronics_entry/rule.md`
on the specific operational rule (RCA ≥ 1.0 across N≥2 codes,
sustained-growth gate), and a
`decisions/2026-04-12_v2-cohort-rule.md` decision record on
why the v2 sustained-growth gate was added.

## The pieces

### 1. The convention file (`.claude/conventions/methods.md`)

Documents the boundary with `decisions/` and
`wiki/concepts/`, the seven required sections, the versioning
pattern, and the cross-link discipline with decisions records.
Read on demand by Claude when the user mentions a cohort rule,
sample restriction, or threshold filter — or asks "where do I
document this method."

### 2. The templates (`templates/methods/{README.md,EXAMPLE_method/rule.md}`)

Seeded by `scr init` into target projects.

- `README.md` — one-liner pointing at the convention.
- `EXAMPLE_method/rule.md` — generic worked example with all
  seven required sections, illustrative content, and a
  diagnostic-counts block in the right shape. Researchers
  replace `EXAMPLE_method/` with their first real method.

The example's content is generic (a tradable-classification
rule with plausible thresholds, not a Cambodia-specific one)
so the framework stays open-source.

### 3. The CLAUDE.md pointer (~5 lines)

Tells Claude that `methods/` exists, names the required
sections, and points at the convention file for the protocol.

### 4. Audit trail (git, not a separate log)

The diagnostic counts in `rule.md` are the headline anchor.
Re-running the implementing script reproduces them; if they
drift, either the rule changed or the data did. To trace the
genealogy:

```bash
git log -- methods/<method>/rule.md            # rule edits
git log -- methods/<method>/                   # everything in the folder
git log --grep 'Run: scripts/build_06_'        # commits that ran the script
```

The `analytical-commit-format` convention's `Run:` and `Out:`
lines tie commits to scripts to outputs.

## Why no `/methods-lint` skill

Same logic as data-sources: useful but trivially failable. A
lint skill would check the seven required sections and the
diagnostic-counts block's freshness. The discipline is the
convention; ship a lint if pilot use shows the rules slip
frequently.

## Why diagnostic counts and not just text descriptions

Considered and rejected: prose-only diagnostic descriptions
("the narrow cohort has roughly 50 entrants"). Rejected
because:

- **Prose loses precision.** "Roughly 50" is uncheckable. "49"
  is.
- **Prose decouples from the script.** A re-run that produces
  47 doesn't surface as a doc-script mismatch if the prose
  says "roughly 50."
- **Prose loses the cross-tab.** "Narrow class breakdown:
  sustainer 39, faller 6, plateauer 2, pending 2" is the kind
  of distribution that catches a regression instantly. Prose
  hides it.

Counts are the anchor.

## Tradeoffs accepted

- **Diagnostic counts go stale.** A rule.md not re-generated
  when data updates carries an out-of-date count. The
  convention asks for a re-run on every load-bearing rule
  edit; the `analytical-commit-format` `Run:` line is the
  discipline. A future `/methods-lint` could check.
- **The seven-section template is rigid.** Some methods don't
  need an Exclusions section (the rule has no exclusions). The
  convention requires the section to be present — even as
  "None; the rule applies universally" — so the doc reads
  consistently across methods. The cost is occasional one-line
  "None" sections; the benefit is that a reader scanning N
  methods doesn't have to figure out which sections were
  elided.
- **Versioning is implicit, not stamped.** The `(v2)` in the
  heading is the only version marker. Researchers needing
  precise vN+1 → vN regeneration use git history and the
  `decisions/` record. We did not build a `versions/` archive
  per method; git suffices.

## Provenance

The convention is a port of the working pattern in
`~/cambodia-growth/methods/electronics_entry/cohort_rule.md`,
which evolved from v1 to v2 mid-engagement when v1 admitted
Cambodia 2013 as an entrant on the basis of three SITC-4 codes
(7591/7599/7649 — "Parts, nes" residual buckets) that fell
back below RCA threshold by 2016. v2 added a sustained-growth
gate; the diagnostic counts dropped from 113 entrants to 49.

The lesson: a methodology rule that admits a country on
residual catch-all codes is broken even if the codes
individually pass the rule. Caught only because the diagnostic
counts at the bottom of the doc surfaced the unexpected count.
Without the counts, the rule would have shipped a chart with
KHM 2013 in it; with the counts, the rule got fixed.

The methods convention codifies the pattern so the next project
catches its analog earlier.
