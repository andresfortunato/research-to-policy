# raw/

Immutable source materials. The archive layer.

## Rules

1. **Never edit a file in `raw/`.** Once written, it stays. If a source is
   wrong, note the correction in the corresponding `wiki/` page — not here.
2. **Always preserve.** Do not delete from `raw/` even if a source seems
   superseded. The full audit trail is part of the value.
3. **Name with a date prefix.** Format: `YYYY-MM-DD_<slug>.<ext>`. The date
   is the day the source was acquired (downloaded, scraped, transcribed),
   not the publication date — publication date belongs in the `wiki/`
   source page's frontmatter.

## What counts as raw

- PDFs of papers, books, reports.
- Scraped articles (full HTML, or markdown-converted-from-HTML).
- Dataset codebooks, README files shipped with downloads.
- Transcripts (interviews, meetings, podcasts).
- Photographs of physical documents.
- Screenshots of dashboards or tables that can't be exported.

If it came from outside the project and you'd want to point at it later as
evidence, it goes here.

## What does NOT count

- Anything *generated by the project* (charts, panel CSVs, regression
  outputs). Those belong in `output/` and are governed by the manifest log.
- Personal notes — those are scratch, not sources.
- Drafts of project deliverables — those go in `deliverables/`.

## Subtree convention

Two layouts coexist under `raw/`:

- **Loose files at the root** — `raw/YYYY-MM-DD_<slug>.<ext>`. Use for
  one-off acquisitions: a paper a colleague forwarded, a dataset
  codebook, an ad-hoc scrape from the `web-scraping` skill, a meeting
  transcript. No registry entry, no recurring fetch.

- **Continuous-scrape subtrees** — `raw/sources/<slug>/YYYY-MM-DD_<title-slug>.md`.
  Use for sources tracked in `sources/registry.yaml` and fetched by
  `/scan-sources`. The skill creates the per-slug directory on first
  fetch; one file per fetched item; YAML frontmatter carries `url`,
  `source_slug`, `category`, `scraped_at`, `content_sha256`, `title`.

The `raw/sources/<slug>/` layout is reserved for `/scan-sources`
output. Don't drop loose files into a slug directory; if a one-off
acquisition relates to a tracked source, put it at the `raw/` root and
note the relationship in the wiki page.

See `.claude/conventions/source-registry.md` for the registry protocol
and `templates/sources/README.md` for how to register a new source.

## Ingest into the wiki

`raw/` is the archive; `wiki/` is the queryable layer. To bring a `raw/`
file into the wiki, run:

```
/wiki-ingest raw/<filename>
```

See `wiki/README.md` and `wiki/SCHEMA.md` for what ingest produces.
